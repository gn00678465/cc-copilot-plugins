# PR 審查指導原則 (zh-TW)

這是用於自動化 PR 審查的技術準則與噪音過濾規則。

---

## 審查面向

### 1. 邏輯與錯誤 (Bugs and Logic)
- **邊界條件**: 陣列索引、空值 (Null/Nil) 檢查。
- **異步處理**: Race conditions、未處理的 Promise。
- **安全性**: SQL Injection 潛在點、未轉義的 HTML (XSS)。

### 2. 程式碼清晰度 (Code Clarity)
- **早回傳 (Early Returns)**: 避免過深的 `if/else` 巢狀結構。
- **命名**: 變更項目是否具有描述性？是否需要心智解碼？
- **簡化**: 是否有不必要的複雜抽象？

### 3. 專案規範 (Project Compliance)
- 是否符合現有專案的 Coding Style？
- 測試覆蓋度是否足夠（針對 Bug Fix 與 Feature）？

---

## 噪音過濾 (Noise Filtering)

為了確保審查報告的價值，請遵循以下原則：

1. **僅回報新增內容**: 不要回報原本就存在於程式碼中，但此次 PR 未修改的舊問題。
2. **信心度過濾**: 僅報告您有把握 (> 80% 信心) 是錯誤或不良實作的項目。
3. **忽略格式細微差別**: 若有 Linter 處理的縮排或空格問題，請予忽略。
4. **具體建議**: 所有的「建議」都必須附帶具體的修正方式或理由。

---

## 嚴重程度定義

- **Critical**: 會造成程式崩潰、資料遺失或重大安全漏洞。
- **Important**: 違反專案規範、引入技術債或明顯影響程式碼維護性。
- **Suggestion**: 可改善清晰度但非錯誤的細項建議。
